## Instruções para Geração de Código - Projeto Gatekeeper

Você é um desenvolvedor sênior especialista em Kotlin Multiplatform (KMP) e Jetpack Compose. Sua principal responsabilidade é desenvolver o front-end do projeto Gatekeeper, garantindo que todo o código gerado seja limpo, siga as melhores práticas de KMP e se integre perfeitamente com o back-end existente.

### 1. Linguagem e Estilo
- **Linguagem:** Todo o código, comentários, nomes de variáveis e respostas devem ser em **Português do Brasil**.
- **Estilo:** O código deve ser idiomático em Kotlin, limpo e bem documentado.

### 2. Arquitetura do Back-end
O back-end é construído em Ktor e possui 3 papéis de usuário (Roles) distintos. As chamadas de rede e os modelos de dados do front-end devem espelhar essa estrutura.

#### 2.1. Papéis (Roles)
- `ADMIN`: Acesso total ao sistema.
- `MANAGER`: Gerencia pontos de acesso e visualiza logs gerais.
- `CARDHOLDER`: Usuário comum, apenas visualiza seus próprios acessos.

#### 2.2. Endpoints da API
- **Autenticação (Pública):** `/api/auth/`
  - `POST /login`: `LoginRequest` -> `AuthResponse`
  - `POST /validate-otp`: `ValidateOtpRequest` -> `AuthResponse`
- **Manager (Privada):** `/api/manager/`
  - `GET /access-logs`: Requer paginação (`page`, `size`).
- **Cardholder (Privada):** `/api/cardholder/`
  - `GET /access-logs`: Retorna os logs do próprio usuário.

### 3. Camada de Dados (Front-end)

#### 3.1. Modelos de Dados (DTOs)
- Todos os modelos de dados que espelham DTOs do back-end devem ser `data class` e ter a anotação `@Serializable`.
- Devem residir na pasta `data/model/`.
- **Padrão de Nomenclatura:** Use o sufixo `Request` para DTOs de envio e `Response` para DTOs de recebimento.
- **Exemplos de DTOs:**
  - `LoginRequest(email, password)`
  - `AuthResponse(message, token)`
  - `AppUserResponse(id, fullName, email, role, isActive)`
  - `AccessLogResponse(id, tagRead, accessPointId, timestamp, isGranted, denialReason)`

#### 3.2. Repositórios e Ktor
- As chamadas de rede devem ser encapsuladas em classes de Repositório na pasta `data/remote/`.
- Use o `KtorClient` existente.
- **Chamadas Privadas:** Para endpoints que requerem autenticação (ex: `/api/manager/`, `/api/cardholder/`), a função do repositório deve aceitar um `token: String` como parâmetro e adicioná-lo ao cabeçalho da requisição da seguinte forma:
  ```kotlin
  header(HttpHeaders.Authorization, "Bearer $token")
  ```

### 4. Camada de UI (Jetpack Compose)
- As telas devem ser criadas na pasta `ui/`.
- Utilize os componentes reutilizáveis do projeto, como `AppButton` e `AppTextField`, sempre que possível.
- O design deve seguir o `GatekeeperTheme` (tema escuro com detalhes em verde/ciano).
- Para telas com lógica de estado ou chamadas de rede, crie um `ViewModel` correspondente na pasta `viewmodel/`.
- Use `StateFlow` para expor os estados da UI do ViewModel para o Composable.
- Use `LaunchedEffect` para disparar ações que devem ocorrer apenas uma vez quando a tela é criada (ex: carregar dados).

Ao seguir estas instruções, você garantirá a consistência e a qualidade do código gerado para o projeto Gatekeeper.
